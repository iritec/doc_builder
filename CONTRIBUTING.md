## 貢献の方法

### 🐛 バグ報告

バグを見つけた場合は、Issue を作成してください：

1. 既存の Issue を検索して、同じバグが報告されていないか確認
2. 新しい Issue を作成し、以下の情報を含める：
   - バグの詳細な説明
   - 再現手順
   - 期待される動作
   - 実際の動作
   - 環境情報（OS、ブラウザ、Node.js バージョンなど）

### 💡 機能リクエスト

新しい機能のアイデアがある場合は、Issue を作成してください：

1. 既存の Issue を検索して、同じ機能が提案されていないか確認
2. 新しい Issue を作成し、以下の情報を含める：
   - 機能の詳細な説明
   - なぜこの機能が必要か（ユースケース）
   - 可能であれば、実装のアイデア

### 🔧 コードの貢献

コードで貢献する場合は、以下の手順に従ってください：

1. Issue を作成するか、既存の Issue にコメントして取り組む意思を表明
2. リポジトリをフォーク
3. 機能ブランチを作成
4. 変更を実装
5. テストを追加・更新（該当する場合）
6. プルリクエストを作成

## 開発環境のセットアップ

### 前提条件

- Node.js 18.x 以上
- npm または yarn または pnpm

### セットアップ手順

```bash
# リポジトリをフォーク後、クローン
git clone https://github.com/your-username/spec-builder.git
cd spec-builder

# 依存関係をインストール
npm install

# 環境変数を設定
cp .env.example .env.local
# .env.local を編集してAPIキーを設定

# 開発サーバーを起動
npm run dev
```

### 便利なコマンド

```bash
# 開発サーバー
npm run dev

# 本番ビルド
npm run build

# リンター
npm run lint
```

## コーディング規約

### TypeScript

- 厳格な型付けを使用する
- `any` 型の使用を避ける
- インターフェースと型エイリアスを適切に使い分ける

### React

- 関数コンポーネントを使用する
- カスタムフックでロジックを分離する
- `'use client'` ディレクティブを適切に使用する

### スタイリング

- Tailwind CSS を使用する
- コンポーネントのスタイルは Tailwind のユーティリティクラスで記述
- 複雑なスタイルは `cn()` ユーティリティを使用して結合

### ファイル構成

- コンポーネントは `src/components/` 以下に配置
- 機能ごとにディレクトリを分ける
- `index.ts` でエクスポートをまとめる

### コミットメッセージ

コミットメッセージは以下の形式を推奨します：

```
<type>: <subject>

<body>
```

**type の例：**
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントの変更
- `style`: コードの意味に影響しない変更（空白、フォーマットなど）
- `refactor`: リファクタリング
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

**例：**
```
feat: AIプロバイダーの切り替え機能を追加

- ヘッダーにプロバイダー選択ドロップダウンを追加
- ClaudeとGeminiの切り替えに対応
```

## プルリクエストの作成

### 作成前のチェックリスト

- [ ] コードがリンターを通過する (`npm run lint`)
- [ ] ローカルで動作確認済み
- [ ] 必要に応じてドキュメントを更新

### プルリクエストの書き方

1. **タイトル**: 変更内容を簡潔に説明
2. **説明**: 以下を含める
   - 変更の概要
   - 関連する Issue（`Fixes #123` など）
   - スクリーンショット（UI の変更がある場合）
   - テスト方法

### レビュープロセス

- プルリクエストはレビューを受けてからマージされます
- レビューコメントには丁寧に対応してください
- 変更が必要な場合は、同じブランチにコミットを追加してください
